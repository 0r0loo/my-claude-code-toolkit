# /refactor — 리팩터링

기존 코드의 구조를 개선한다. 기능 추가(/feature)나 버그 수정(/fix)이 아닌, 동작을 유지하면서 코드 품질을 높이는 작업.

## 인자
- `[대상]`: 리팩터링 대상 (파일, 모듈, 패턴 등)

## 실행 절차

1. **현재 상태 파악**
   - 대상 코드를 읽고 현재 구조 파악
   - 관련 테스트가 있는지 확인
   - **기존 테스트가 있으면 먼저 실행하여 통과 확인**

2. **리팩터링 계획 제시**
   - 무엇을 왜 바꾸는지 설명
   - 변경 범위 (파일, 함수 목록)
   - **사용자 승인 후 진행**

3. **리팩터링 실행**
   - 한 번에 하나의 변경 (함수 추출, 이름 변경, 구조 분리 등)
   - 단계별로 기존 테스트 통과 확인

4. **검증**
   - 기존 테스트 전체 재실행
   - 동작 변경이 없는지 확인

## 리팩터링 유형

| 유형 | 예시 | 주의점 |
|------|------|--------|
| 추출 (Extract) | 긴 함수 → 작은 함수들 | 함수 경계에서 인터페이스 명확히 |
| 이동 (Move) | 잘못된 위치의 로직 → 적절한 모듈 | import 경로 전부 업데이트 |
| 이름 변경 (Rename) | 의도가 불명확한 이름 → 명확한 이름 | 모든 참조를 빠짐없이 변경 |
| 단순화 (Simplify) | 복잡한 조건 → 가드 절, 얼리 리턴 | 동작이 동일한지 엣지 케이스 확인 |
| 중복 제거 (Deduplicate) | 반복 코드 → 공통 유틸 | 모든 사용처에서 동일하게 동작하는지 확인 |

## 규칙
- **동작을 바꾸지 않는다.** 리팩터링의 정의. 동작이 바뀌면 /feature 또는 /fix.
- **테스트가 없으면 먼저 테스트를 추가한 후 리팩터링한다.**
- **한 번에 하나.** 여러 리팩터링을 섞지 않는다.
- 리팩터링과 기능 변경을 같은 커밋에 넣지 않는다.
