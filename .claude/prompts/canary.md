배포 후 프로덕션 상태를 검증한다. 지정 URL에 접속하여 에러, 성능, 기본 동작을 체크한다.

## 입력
- $ARGUMENTS: 검증할 URL (필수)

---

## 절차

### Phase 1 — 기본 접근성 확인
1. 지정 URL에 HTTP 요청을 보낸다
2. 응답 상태 코드를 확인한다 (200 OK 기대)
3. 응답 시간을 측정한다
4. HTTPS 리다이렉트가 정상 동작하는지 확인한다

### Phase 2 — 페이지 검증
1. 페이지 콘텐츠가 정상적으로 로드되는지 확인한다
2. 주요 API 엔드포인트가 응답하는지 확인한다 (있는 경우)
3. 콘솔 에러 패턴을 탐지한다:
   - JavaScript 에러 (TypeError, ReferenceError 등)
   - 네트워크 에러 (404, 500 리소스)
   - CORS 에러
   - Mixed Content 경고

### Phase 3 — 성능 기본 체크
1. 응답 시간이 임계값을 초과하는지 확인한다
   - 양호: < 1초
   - 주의: 1~3초
   - 위험: > 3초
2. 주요 리소스(JS, CSS, 이미지) 로드 실패가 있는지 확인한다

### Phase 4 — 카나리 리포트

```
## Canary Report — [URL]
시각: [현재 시각]

### 접근성
- 상태 코드: [200/etc]
- 응답 시간: [N]ms
- HTTPS: [정상/비정상]

### 페이지 검증
- 콘텐츠 로드: [정상/비정상]
- 콘솔 에러: [N]건
  - [에러 목록 (있으면)]
- 네트워크 에러: [N]건
  - [실패 리소스 목록 (있으면)]

### 성능
- 응답 시간 판정: [양호/주의/위험]
- 리소스 로드 실패: [N]건

### 판정: HEALTHY / WARNING / CRITICAL
[사유 한 줄 요약]
```

판정 기준:
| 조건 | 판정 |
|------|------|
| 상태 200, 에러 0, 응답 < 1초 | **HEALTHY** |
| 상태 200, 경고성 이슈 있음 | **WARNING** |
| 상태 비정상 또는 심각한 에러 | **CRITICAL** |

---

## 예시

### /canary https://myapp.com
프로덕션 URL 헬스체크

### /canary https://staging.myapp.com
스테이징 환경 검증

### /canary http://localhost:3000
로컬 배포 후 검증
