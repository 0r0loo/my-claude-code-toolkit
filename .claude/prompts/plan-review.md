# /plan-review — 구현 전 계획 검증

코드를 쓰기 전에 계획을 검증한다. 아키텍처, 데이터 흐름, 테스트 전략, 실패 시나리오를 점검하고 이슈별로 합의한다.

## 인자
- (없음): 현재 대화의 계획을 검증
- `[파일 경로]`: plan.md 또는 디자인 문서를 검증

## 절차

### Phase 1 — 계획 파악

1. 계획 문서 또는 대화 맥락에서 다음을 추출한다:
   - 무엇을 만들려는가?
   - 어떤 파일/모듈이 변경되는가?
   - 어떤 의존성이 있는가?
2. 변경 범위를 파악한다: 파일 수, 새 추상화 수

### Phase 2 — 5가지 차원 검증

각 차원에서 이슈가 있으면 **하나씩** 질문하고 선택지를 제시한다. 이슈 없으면 PASS.

#### 1. 범위 (Scope)
- 이게 정말 필요한 전부인가? 빠진 게 없는가?
- 반대로, 과도한 게 없는가? 지금 안 해도 되는 게 섞여 있는가?
- **가치관 체크**: "가장 단순한 버전"인가?

#### 2. 아키텍처 (Architecture)
- 컴포넌트 경계가 적절한가?
- 의존성 방향이 맞는가? 순환 의존은 없는가?
- 기존 코드/패턴을 재사용하는가, 불필요하게 새로 만드는가?
- 파일 8개 이상 또는 새 추상화 2개 이상이면 → 경고: "더 단순하게 할 수 없는가?"

#### 3. 데이터 흐름 (Data Flow)
- 데이터가 어디서 생성되어 어디로 흐르는가? (ASCII 다이어그램)
- 병목 지점이 있는가?
- 단일 장애점(SPOF)이 있는가?

#### 4. 실패 시나리오 (Failure Modes)
- 각 새 코드패스에서 현실적인 실패 1가지를 식별
- 에러 핸들링이 계획에 포함되어 있는가?
- 사용자에게 명확한 에러가 보이는가, 조용히 실패하는가?
- **가치관 체크**: "되돌릴 수 있는가?"

#### 5. 테스트 전략 (Test Strategy)
- 새 코드패스마다 테스트가 계획되어 있는가?
- 어떤 레벨의 테스트가 필요한가? (단위/통합/E2E)
- 엣지 케이스는 식별되었는가?

### Phase 3 — 이슈 합의

이슈별로 하나씩:
```
🔍 이슈 [N]: [제목]

[문제 설명 — 기술 용어 최소화, 구체적으로]

선택지:
A) [수정안] — [장단점]
B) [대안] — [장단점]
C) 그대로 진행 — [리스크]

추천: [A/B/C] — [이유]
```

모든 이슈가 합의되면 Phase 4로.

### Phase 4 — 결과 보고

```
## /plan-review 결과

범위:        PASS / [수정 내용]
아키텍처:    PASS / [수정 내용]
데이터 흐름: PASS / [수정 내용]
실패 시나리오: PASS / [수정 내용]
테스트 전략: PASS / [수정 내용]

이슈: [N]건 발견, [M]건 합의
판정: APPROVED / NEEDS_REVISION
```

APPROVED면 → 구현 시작 가능 (다음: `/feature` 또는 implementer 에이전트)

## 규칙
- **코드를 작성하지 않는다.** 계획 검증만 한다.
- **이슈는 하나씩 질문한다.** 한꺼번에 5개 던지지 않는다.
- **구체적으로 문제를 말한다.** "고려해보세요"가 아니라 "이 부분에서 N+1 쿼리가 발생할 수 있습니다"
- **과도하게 막지 않는다.** 모든 계획에 이슈를 찾으려 하지 말고, 진짜 문제만 짚는다.
- M/L 티어 작업에서 구현 전 실행을 권장한다.
