지정 기간의 작업을 회고한다. git 히스토리 기반으로 작업 요약, 트렌드, 개선점을 분석한다.

## 입력
- $ARGUMENTS가 없으면 → 최근 1주 (7일)
- $ARGUMENTS가 있으면 → 지정 기간 (`7d`, `14d`, `30d`, `2w`, `1m`)

---

## 절차

### Phase 1 — 데이터 수집
1. `git log --since="[기간]" --oneline --format="%h %s (%ai)"` 으로 커밋 목록 수집
2. `git log --since="[기간]" --shortstat` 으로 변경량 수집
3. 브랜치별, 커밋 메시지 접두사별(FEAT/FIX/DOCS/REFACTOR) 분류

### Phase 2 — 분석
1. **작업 요약**: 기간 내 주요 변경사항을 카테고리별로 정리
2. **수치 요약**: 커밋 수, 파일 변경 수, 추가/삭제 라인
3. **트렌드 분석**:
   - 기능 추가 vs 버그 수정 비율
   - 가장 많이 변경된 파일/디렉토리
   - 커밋 빈도 패턴 (요일별, 시간대별)
4. **잠재 리스크**:
   - 핫스팟 파일 (짧은 기간에 반복 수정된 파일)
   - 대규모 변경 커밋 (100+ lines 단일 커밋)
   - 테스트 없는 기능 커밋

### Phase 3 — 이전 회고와 비교

`.claude/retro-history.json`이 있으면 이전 회고 데이터와 비교한다.

1. 이전 회고의 수치와 현재를 비교:
   - 기능:버그 비율 변화 (개선 ↑ / 악화 ↓)
   - 핫스팟 파일이 해소되었는지 또는 지속 중인지
   - 커밋 빈도 변화
2. 이전 회고의 "다음 주기 제안"이 실행되었는지 확인
3. 파일이 없으면 이 단계를 건너뛴다

### Phase 4 — 회고 리포트

```
## Retro Report — [시작일] ~ [종료일]

### 요약
- 커밋: [N]개 | 파일 변경: [N]개 | +[N] / -[N] lines

### 주요 작업
#### FEAT
- [커밋 메시지 요약]

#### FIX
- [커밋 메시지 요약]

#### DOCS / REFACTOR / 기타
- [커밋 메시지 요약]

### 트렌드
- 기능:버그 비율: [N:M] (이전: [N:M] → 변화: ↑/↓)
- 핫스팟 파일: [가장 많이 변경된 파일 Top 3]
- 활동 패턴: [요일/시간대 특이사항]

### 이전 회고 대비 (히스토리가 있을 때만)
- ✅ 해소됨: [이전 핫스팟/리스크 중 해결된 것]
- ⚠️ 지속 중: [이전에도 있었고 여전한 것]
- 🆕 새로 발생: [이번에 처음 나타난 것]

### 잠재 리스크
- [핫스팟, 대규모 커밋, 테스트 누락 등]

### 다음 주기 제안
- [개선점, 주의사항, 우선순위 제안]
```

### Phase 5 — 히스토리 저장

현재 회고 데이터를 `.claude/retro-history.json`에 추가한다.

```json
{
  "history": [
    {
      "date": "2026-03-22",
      "period": "7d",
      "commits": 15,
      "feat_fix_ratio": "3:1",
      "hotspots": ["install.sh", "CLAUDE.md"],
      "risks": ["install.sh 반복 수정"],
      "suggestions": ["테스트 커버리지 추가"]
    }
  ]
}
```

최근 10건만 유지한다 (오래된 항목 자동 삭제).

## 예시

### /retro
최근 7일 회고

### /retro 14d
최근 14일 회고

### /retro 1m
최근 1개월 회고
