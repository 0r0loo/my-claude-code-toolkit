# /simplify — 변경 코드 품질 점검

구현 직후, 리뷰 전에 실행하는 자가 점검. 변경된 코드의 재사용성, 품질, 효율성을 검토하고 문제가 있으면 직접 수정한다.

## 실행 절차

1. **변경 파일 파악**
   ```bash
   git diff --name-only HEAD~1
   ```
   변경된 파일이 없으면 staged 또는 unstaged 변경을 확인한다.

2. **3가지 관점으로 검토**

   ### 재사용성 (Reuse)
   - 같은 로직이 2곳 이상에 반복되는가?
   - 기존 유틸/헬퍼로 대체 가능한 코드가 있는가?
   - 하드코딩된 값이 상수/설정으로 빠져야 하는가?

   ### 품질 (Quality)
   - 네이밍이 의도를 드러내는가?
   - 에러 핸들링이 누락된 경로가 있는가?
   - 타입이 느슨한 곳이 있는가? (any, as, 빈 제네릭)
   - 미사용 import/변수가 있는가?

   ### 효율성 (Efficiency)
   - 불필요한 재렌더링/재계산이 있는가?
   - N+1 쿼리나 루프 안 비동기 호출이 있는가?
   - 과도한 추상화가 있는가? (한 번만 쓰이는 헬퍼, 불필요한 래퍼)

3. **문제 발견 시 직접 수정**
   - 각 문제를 한 줄로 설명하고 즉시 수정
   - 수정이 범위를 넘어서면 (다른 모듈 변경 필요 등) 수정하지 않고 보고만

4. **결과 보고**
   ```
   ## /simplify 결과
   - ✅ 재사용성: 문제 없음
   - 🔧 품질: [수정한 내용]
   - ✅ 효율성: 문제 없음

   수정: N건 / 보고만: M건
   ```

## 규칙
- **과도하게 고치지 않는다.** 변경된 파일만 본다. 주변 코드를 리팩터하지 않는다.
- **스타일 변경은 하지 않는다.** 포매터/린터의 영역은 건드리지 않는다.
- **기능 변경은 하지 않는다.** 동작이 바뀌는 수정은 /simplify 범위가 아니다.
