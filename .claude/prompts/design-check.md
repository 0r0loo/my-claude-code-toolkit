# /design-check — 구현 전 UI/UX 설계 검증

코드를 쓰기 전에 UI/UX 설계를 점수화하고, 각 차원에서 10점이 되려면 뭘 해야 하는지 구체적으로 제안한다.

## 인자
- (없음): 현재 대화의 UI 설계를 검증
- `[파일 경로]`: 와이어프레임, 디자인 문서 검증

## 절차

### Phase 1 — 설계 파악

1. 설계 문서, 와이어프레임, 또는 대화에서 UI 계획을 추출한다
2. 어떤 화면/컴포넌트를 만들려는지 파악한다
3. DESIGN.md가 있으면 디자인 시스템 제약을 참고한다

### Phase 2 — 6차원 점수화

각 차원을 0-10으로 평가하고, **10점이 되려면 뭘 해야 하는지** 구체적으로 말한다.

#### 1. 정보 계층 (Hierarchy) — /10
- 사용자가 화면을 봤을 때 첫 번째, 두 번째, 세 번째로 보는 게 뭔가?
- 중요도 순서가 시각적으로 명확한가?
- 10점이 되려면: [구체적 제안]

#### 2. 인터랙션 상태 (States) — /10
- 로딩 상태가 있는가?
- 빈 상태 (데이터 없을 때)가 있는가?
- 에러 상태가 있는가?
- 성공 피드백이 있는가?
- 10점이 되려면: [구체적 제안]

#### 3. 레이아웃 & 간격 (Layout) — /10
- 간격이 일관적인가? (4px/8px 배수 등)
- 정렬이 맞는가?
- 여백이 적절한가? (너무 빡빡하거나 너무 넓지 않은가)
- 10점이 되려면: [구체적 제안]

#### 4. 반응형 (Responsive) — /10
- 모바일 (375px)에서 어떻게 보이는가?
- 태블릿 (768px)에서는?
- 콘텐츠가 넘칠 때 어떻게 처리하는가?
- 10점이 되려면: [구체적 제안]

#### 5. 접근성 (Accessibility) — /10
- 색상 대비가 WCAG AA를 충족하는가?
- 키보드로 조작 가능한가?
- 스크린 리더에 의미 있는 텍스트가 전달되는가?
- 터치 타겟이 44px 이상인가?
- 10점이 되려면: [구체적 제안]

#### 6. 엣지 케이스 (Edge Cases) — /10
- 텍스트가 아주 길면? (47자 이름 등)
- 데이터가 0건이면? 1000건이면?
- 네트워크가 느리면?
- 권한이 없는 사용자가 보면?
- 10점이 되려면: [구체적 제안]

### Phase 3 — 이슈별 합의

7점 미만인 차원에 대해 하나씩 개선 방안을 제시하고 합의한다:

```
🎨 [차원명]: [N]/10 → 목표 [M]/10

현재 문제: [구체적 설명]

개선안:
A) [제안] — 10점 가능
B) [가벼운 대안] — 8점 가능
C) 그대로 — 현재 점수 유지

추천: [A/B/C]
```

### Phase 4 — 결과 보고

```
## /design-check 결과

| 차원 | 점수 | 변경 |
|------|------|------|
| 정보 계층 | [N]/10 | [변경 여부] |
| 인터랙션 상태 | [N]/10 | [변경 여부] |
| 레이아웃 & 간격 | [N]/10 | [변경 여부] |
| 반응형 | [N]/10 | [변경 여부] |
| 접근성 | [N]/10 | [변경 여부] |
| 엣지 케이스 | [N]/10 | [변경 여부] |

전체 평균: [N]/10
판정: APPROVED / NEEDS_REVISION
```

APPROVED (평균 7점 이상) → 구현 시작 가능

## 규칙
- **코드를 작성하지 않는다.** 설계 검증만 한다.
- **점수에 근거를 댄다.** "8점입니다"가 아니라 "로딩 상태가 없어서 8점입니다"
- **10점이 되려면 뭘 할지 항상 말한다.** 점수만 매기면 가치가 없다.
- **UI가 없는 작업이면 이 커맨드를 쓰지 않는다.** 백엔드만이면 `/plan-review`로.
- 7점 미만이 없으면 전체 PASS — 빠르게 끝낸다.
