버그/이슈를 조사한다. **조사 없이 수정 금지.**

## 입력
- 증상: $ARGUMENTS

## 원칙
수정 전에 반드시 근본 원인을 확정한다.
- 가설 단계에서 코드를 수정하지 않는다
- "아마 이게 문제일 것 같다"는 확정이 아니다
- 재현이 불가능하면 수정도 불가능하다

## 절차

### Phase 1 — 재현 (Reproduce)
1. 증상을 정확히 기술한다 (에러 메시지, 발생 조건, 빈도)
2. 재현 조건을 좁힌다:
   - 어떤 입력/환경에서 발생하는가?
   - 항상 발생하는가, 간헐적인가?
   - 특정 데이터/상태에서만 발생하는가?
3. 재현 불가 시 → 사용자에게 추가 정보 요청 후 중단

### Phase 2 — 원인 확정 (Diagnose)
1. 관련 코드를 탐색한다 (built-in Explore 또는 Grep/Glob 활용)
   - **탐색 범위를 증상과 연관된 모듈로 제한한다**
   - 관계없는 코드는 읽지 않는다
2. 가설 목록을 작성한다 (최대 3개)
3. 각 가설을 코드 근거로 검증한다
4. 근본 원인 1개를 확정한다 — 확정되지 않으면 추가 탐색

### Phase 3 — 보고서 작성 (Report)
아래 형식으로 investigation-report를 출력한다.
수정은 이 단계에서 하지 않는다.

## 출력: Investigation Report

```
## Investigation Report

### 증상 (Symptom)
- 발생 조건: ...
- 에러 메시지: ... (없으면 "없음")
- 재현 여부: 재현됨 / 간헐적 / 재현 불가

### 재현 조건 (Reproduction)
- 입력/환경: ...
- 재현 단계:
  1. ...
  2. ...

### 가설 목록 (Hypotheses)
1. [기각/채택] 가설 — 근거: [파일:라인] ...
2. [기각/채택] 가설 — 근거: [파일:라인] ...
3. [기각/채택] 가설 — 근거: [파일:라인] ...

### 근본 원인 (Root Cause)
[파일:라인] 구체적인 원인 설명
- 왜 이 코드가 문제인가: ...
- 어떤 조건에서 트리거되는가: ...

### 영향 범위 (Impact)
- 영향받는 파일: ...
- 동일 패턴이 다른 곳에도 있는가: 있음 / 없음
  - (있으면 파일 목록 나열)

### 권장 수정 방향 (Recommended Fix)
- 접근 방식: ...
- 수정 범위: S / M / L
- 주의사항: ... (부작용, 테스트 필요 항목)
- /fix 연계: `/fix [근본 원인 한 줄 요약]`
```

## /fix 연계

investigation-report 완료 후 수정이 필요하면:
```
/fix [investigation-report의 "근본 원인" 내용]
```
`/fix`는 이 보고서의 "권장 수정 방향"을 입력으로 사용한다.
수정 범위는 "영향 범위"에 명시된 파일로 제한한다.

## 예시

### /investigate 저장 버튼 클릭 시 간헐적으로 500 에러

**Phase 1**: POST /api/posts 요청이 간헐적으로 500 반환. 특정 제목 길이(> 100자)에서 재현됨.

**Phase 2**:
- 가설 1: 제목 길이 validation 누락 → posts.controller.ts:42 확인 → `@MaxLength` 없음 → 채택
- 가설 2: DB 커넥션 풀 고갈 → DB 로그 확인 → 정상 → 기각

**Phase 3**: investigation-report 출력 후 `/fix posts.controller.ts의 title 필드에 MaxLength validation 누락`
