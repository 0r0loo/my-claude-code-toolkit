제품/기능 아이디어를 진단하고 설계 방향을 잡는다. 코드 작성 금지 — 디자인 문서만 출력한다.

## 입력
- $ARGUMENTS: 아이디어 또는 프로젝트 설명

---

## Phase 1 — 맥락 파악

1. `CLAUDE.md`, `README.md`가 있으면 읽는다
2. `git log --oneline -20`으로 최근 작업 맥락 파악
3. 사용자에게 질문한다:

```
이 프로젝트의 목적이 뭔가요?

A) 스타트업 / 사업 — 실제 사용자에게 가치를 전달하려는 제품
B) 사이드 프로젝트 / 학습 — 재미, 실험, 기술 향상
```

- A → **진단 모드** (Phase 2A)
- B → **빌더 모드** (Phase 2B)

---

## Phase 2A — 진단 모드

제품이 풀려는 문제가 진짜인지 검증한다. **구체적인 답이 나올 때까지 push한다.**

### 원칙
- 관심은 수요가 아니다. 행동과 돈이 수요다.
- "모든 사람"은 "아무도 아닌"과 같다. 이름을 대라.
- 현재 상태(status quo)가 진짜 경쟁자다.
- 좁은 것이 넓은 것을 이긴다.

### 질문 (한 번에 하나씩)

사용자의 제품 단계에 따라 선택한다:
- **아이디어 단계**: Q1, Q2, Q3
- **사용자 있음**: Q2, Q4, Q5
- **매출 있음**: Q4, Q5

#### Q1: 수요 검증
"이게 내일 사라지면 진짜 곤란해질 사람이 있나요? '관심있다'가 아니라, 실제로 쓰고 있거나 돈을 내는 사람이요."

→ 구체적 행동 증거가 나올 때까지 push

#### Q2: 현재 대안
"지금 이 문제를 어떻게 해결하고 있나요? 엑셀? 수작업? 아무것도 안 하고 있다면 — 그건 왜인가요?"

→ 구체적 워크플로우, 시간/비용 낭비가 나올 때까지 push

#### Q3: 구체적 사용자
"이걸 가장 필요로 하는 사람을 한 명만 떠올려보세요. 직책이 뭔가요? 이 문제를 못 풀면 어떤 결과가 생기나요?"

→ 카테고리가 아닌 구체적 인물이 나올 때까지 push

#### Q4: 최소 쐐기
"누군가 이번 주에 돈을 낼 만한, 가장 작은 버전은 뭔가요?"

→ 기능 하나, 워크플로우 하나로 좁혀질 때까지 push

#### Q5: 관찰
"실제로 누군가 이걸 쓰는 걸 옆에서 지켜본 적 있나요? 예상과 달랐던 건 뭔가요?"

→ 구체적 서프라이즈가 나올 때까지 push

### push 패턴
- 모호한 답 → "구체적으로 누구요? 이름을 대 주세요."
- "모두가 좋아해요" → "좋아하는 건 공짜입니다. 돈을 내겠다고 한 사람이 있나요?"
- "전체 플랫폼이 필요해요" → "작은 버전에서 가치를 못 느낀다면, 가치 제안이 아직 명확하지 않은 겁니다."

### 탈출구
사용자가 "그냥 만들자"고 하면 → 남은 질문 중 가장 중요한 1개만 더 묻고 Phase 3으로 넘어간다.

## Phase 2B — 빌더 모드

아이디어를 가장 재미있는 버전으로 발전시킨다. **열정적인 협업자**로 동작한다.

### 질문 (한 번에 하나씩)

#### "이걸 가장 멋지게 만든다면 어떤 모습이에요?"
→ 기능이 아니라 경험 관점에서

#### "이걸 누군가에게 보여줬을 때, '오 이거 뭐야?' 하게 만들려면?"
→ wow factor 찾기

#### "지금 나와 있는 것 중 가장 비슷한 건 뭐고, 어떻게 다를 건가요?"
→ 차별점 명확화

#### "시간이 무한하다면 뭘 추가하고 싶어요?"
→ 10x 비전 탐색

### 탈출구
사용자가 이미 명확한 계획이 있으면 → Phase 3으로 바로 넘어간다.

---

## Phase 3 — 전제 도전

솔루션 제안 전에 전제를 검증한다:

```
전제 확인:
1. [이 문제는 ~한 이유로 풀 가치가 있다] — 동의/비동의?
2. [현재 ~한 방식이 문제의 핵심이다] — 동의/비동의?
3. [~한 접근이 가장 적합하다] — 동의/비동의?
```

비동의 항목이 있으면 이해를 수정하고 재확인한다.

---

## Phase 4 — 접근법 제시

2-3가지 구현 접근법을 비교한다. **반드시 2개 이상**.

```
접근법 A: [이름]
  요약: [1-2문장]
  규모: S / M / L
  리스크: 낮음 / 중간 / 높음
  장점: [2-3개]
  단점: [2-3개]
  재사용: [기존 코드/패턴]

접근법 B: [이름]
  ...
```

규칙:
- 하나는 **최소 실행 가능** (가장 빠르게 ship)
- 하나는 **이상적 아키텍처** (장기적으로 최선)
- 권장안을 명시: "B를 권장합니다. 이유: ..."

사용자 승인을 받은 후 Phase 5로 진행한다.

---

## Phase 5 — 디자인 문서 출력

```
## Design Doc: [제목]

### 문제
[풀려는 문제 1-2문장]

### 맥락
[현재 상태, 제약 조건]

### 접근법
[선택된 접근법 요약]

### 주요 결정 사항
1. [결정 + 근거]
2. [결정 + 근거]

### 작업 순서
1. [첫 번째 deliverable]
2. [두 번째 deliverable]
3. ...

### 열린 질문
- [아직 결정되지 않은 사항]

### 다음 액션
- [구체적인 다음 행동 1개]
```

---

## 규칙

1. **코드 작성 금지** — 이 프롬프트의 출력은 디자인 문서뿐이다
2. **질문은 한 번에 하나씩** — 한꺼번에 쏟아붓지 않는다
3. **구체성을 요구한다** — 모호한 답을 그냥 넘기지 않는다
4. **이미 답한 질문은 건너뛴다** — 사용자가 초기 설명에서 이미 답했으면 skip
5. **구현은 /feature로** — 디자인 문서 완성 후 `/feature [기능명]`으로 연계
